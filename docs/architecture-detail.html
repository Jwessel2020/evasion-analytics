Evasion v3.3.0

Multi-task FT-Transformer for tabular police enforcement prediction

Trained on 1.27M historical Montgomery County Police stops with walk-forward 5-fold CV (skip fold 2 = degenerate). PLR tokenizer + leak-safe target encoding + fold-aware citation rate. 01Overview

At a glance — what the model is, what it predicts, and why it matters.

What it is

  • • Architecture: Feature Tokenizer Transformer (FT-T) — Gorishniy 2021
  • • Multi-task: 1 shared backbone, 8 task heads
  • • Tokenizer: Periodic-Linear (PLR) for numerics — Gorishniy 2022
  • • Params: 1,136,160
  • • Inputs: ~109 stop-numeric + 6 categorical embeddings
  • • Validation: walk-forward 5-fold (skip fold 2), 4-fold ensemble

What it predicts

  • • occurrence — expected stops per cell×hour×DOW (Poisson)
  • • speed_occurrence — mobile-detected speed stops (Poisson)
  • • trap — stationary-detected speed stops (Poisson)
  • • speed — P(speed-related | stop) (Binary)
  • • search — P(vehicle search | stop) (Binary)
  • • accident — P(crash | stop) (Binary)
  • • injury — P(injury | stop) (Binary)
  • • disposition — P(citation vs warning | stop) (Binary, NEW v3.3.0)
Composite score: +0.883 across 4 ensembled folds — +14% over previous v3.2.0 and +0.883 over the XGBoost baseline.

04Model Architecture

FT-Transformer with PLR numeric tokenization, 4 transformer blocks, 8 task heads.

Component breakdown

PLR Tokenizer

For each numeric feature x, produce [x, sin(2πW₁x), cos(2πW₁x), ..., sin(2πWₖx), cos(2πWₖx)] with k=8 learned frequencies, then linear-project to 128-d. Captures cyclic structure (hour, longitude) better than raw linear projection.

Categorical Embeddings

Each categorical gets nn.Embedding(N, 128). vehicle_make (3,760 unique values) gets a 481K-param embedding table that the model learns end-to-end — similar makes converge to similar vectors.

Transformer Block

Standard pre-norm transformer: multi-head self-attention (8 heads, d_token=128) → residual → FFN (ReGLU activation) → residual. 4 blocks stacked. Self-attention lets each feature "look at" all others to learn interactions.

[CLS] Token

Special learnable token prepended to feature sequence. After 4 transformer blocks, its 128-d output is the dataset-level summary — the input every task head reads from.

06Prediction Heads

Each head reads from the same shared [CLS] embedding. 3 Poisson + 5 binary.

Head Type Granularity What it predicts Pos rate Loss Weight
occurrence Poisson cell × hour × DOW Expected total stops in this bin — poisson_nll(log_input=True) 0.25
speed_occurrence Poisson cell × hour × DOW Speed stops detected by mobile enforcement — poisson_nll(log_input=True) 0.10
trap Poisson cell × hour × DOW Speed stops detected by stationary radar/laser — poisson_nll(log_input=True) 0.05
speed Binary per stop P(stop is speed-related) 22% BCE(pos_weight=3.6) 0.08
search Binary per stop P(vehicle search conducted) 6% BCE(pos_weight=15) 0.13
accident Binary per stop P(stop linked to accident) 5% BCE(pos_weight=18) 0.18
injury Binary per stop P(personal injury) 2% BCE(pos_weight=48 → cap 50) 0.13
disposition NEW v3.3.0 Binary per stop P(citation vs warning) 64% BCE(pos_weight=0.58) 0.08
Loss weight design. The 8 weights sum to 1.00. Empirical hand-tuning: harder heads (occurrence, accident) get higher weights. disposition (NEW) was rebalanced from speed_occ + speed to make room without hurting other heads.

Why partition speed_occurrence vs trap?

Both are filtered to is_speed_related == 1 first, then partitioned by detection method: speed_occurrence = mobile enforcement codes (A/B/C/D/I/J), trap = stationary codes (radar/laser/VASCAR). Pre-partition the heads were 95% correlated (useless ensemble); post-partition only 5.4% correlated — making them genuinely informative.

07Losses & Composite Metric

How we combine 8 head losses for training and how we measure success.

Total training loss

loss_total = LOSS_WEIGHTS["occurrence"]       * loss_occurrence
           + LOSS_WEIGHTS["speed_occurrence"] * loss_speed_occ
           + LOSS_WEIGHTS["trap"]             * loss_trap
           + LOSS_WEIGHTS["speed"]            * loss_speed
           + LOSS_WEIGHTS["search"]           * loss_search
           + LOSS_WEIGHTS["accident"]         * loss_accident
           + LOSS_WEIGHTS["injury"]           * loss_injury
           + LOSS_WEIGHTS["disposition"]      * loss_disposition

Composite evaluation metric

The single number we use to compare model versions. Centered at 0 = matches XGBoost baseline.

xgb_mae = {"occ_mae": 0.339, "speed_occ_mae": 0.086, "trap_mae": 0.091}
xgb_auc = {"speed_auc": 0.842, "search_auc": 0.883,
           "accident_auc": 0.894, "injury_auc": 0.881,
           "disposition_auc": 0.65}  # baseline estimate

mae_score = sum((xgb_mae[k] - metrics[k]) / xgb_mae[k] for k in xgb_mae)
auc_score = sum((metrics[k] - xgb_auc[k]) for k in xgb_auc)
composite = mae_score + auc_score
  • For MAE heads: relative improvement (positive = lower MAE than XGB = better)
  • For AUC heads: absolute improvement (positive = higher AUC than XGB = better)
  • 0 = matches XGBoost baseline across the board
  • +1.0 ≈ "average of 8 heads each beat XGB by ~12.5%"

08Training Pipeline

Walk-forward CV + Modal multi-container parallel training.

Walk-forward 5-fold CV

For temporal data, random k-fold leaks future into past. We use chronological splits where each fold's training data is strictly BEFORE its test period.

Modal multi-container deployment

The launcher fans out 4 parallel containers via .starmap():

@app.function(
    gpu="L4",
    max_containers=4,    # KEY: allows 4 parallel L4 containers
    timeout=10800,
    volumes={"/features": features_vol, "/checkpoints": checkpoints_vol},
)
def train_one_v330_fold_multi(fold_idx, epochs=60, ...):
    train_fold(args, fold_idx)   # next_gen.train

@app.local_entrypoint()
def main(folds="0,1,3,4", epochs=60, ...):
    args_iter = [(f, epochs, ...) for f in fold_list]
    results = list(train_one_v330_fold_multi.starmap(args_iter))

Why multi-container vs other strategies

StrategyWall timeCostTrade-off
Sequential (1 GPU) 8.4 hr $4.25 Simple, slowest
Packed (4 jobs share 1 GPU) 5 hr $4.00 Cheapest; CUDA serializes kernels
Multi-container (4 GPUs) 2.1 hr $4.25 Chosen — same cost as sequential, 4× faster wall

Per-fold solo VRAM: 1.84 GB. L4 has 24 GB. Comfortable headroom.

Per-fold checkpoint outputs

For each fold, training saves:

  • unified_phase2_fold{N}.pt — last-epoch composite ckpt
  • unified_phase2_fold{N}_best_speed_auc.pt — best speed_auc across all epochs
  • ...same for search_auc, accident_auc, injury_auc, disposition_auc, occ_mae, speed_occ_mae, trap_mae

So 9 checkpoint files per fold × 4 folds = 36 files (~150 MB total). Ensemble inference uses the per-head specialists for each head when available.

09Ensemble Inference

After training, average per-fold predictions on the laptop A4000.

Binary head ensemble (per stop)

for fold in [0, 1, 3, 4]:
    enable_te, leak_safe, expected_n = _decide_te_for_fold(fold)
    stop_ds = build_fold_dataset(stops, bins, fold,
                                  enable_te=enable_te, leak_safe=leak_safe)
    for head in BINARY_HEADS:
        ckpt = load_specialist_ckpt(fold, head)
        probs[head] += sigmoid(ckpt.forward(stop_ds))
probs[head] /= len(folds)   # ensemble average

10The Leak War Story

When AUC = 1.000, you have a leak — twice over.

Symptom: Smoke test of v3.3.0 disposition head showed disposition_auc = 1.000 across all 5 epochs. Too perfect.

Diagnosis #1 — Per-cell citation_rate aggregation leak

What was wrong: pipelines/09j_precompute_cell_lens_stats.py computed per-cell citation_rate = n_citations / (n_citations + n_warnings) over all stops, no fold split. For each stop, the cell rate INCLUDED that stop's own outcome.

The fix: Made 09j accept --fold-idx N flag. When set, filters input to that fold's train+val stops before groupby. Produces cell_lens_stats_fold{N}.parquet for each fold. At training time, load_fold_split swaps the canonical citation_rate column with the right fold-N variant.

Re-smoke result: Still 1.000. There was a SECOND leak.

Diagnosis #2 — violation_type target encoding leak

What was wrong: TE_CATEGORICALS in deep/data.py contained violation_type (4 values: Citation, Warning, ESERO, SERO). For each (cat, target) pair, runtime TE produced one column. So violation_type_te_is_speed_related, violation_type_te_search_conducted, violation_type_te_accident, violation_type_te_personal_injury — 4 columns from violation_type alone.

Each violation_type value gets a UNIQUE 4-tuple of TE values across those 4 columns. The disposition head sees those 4 columns and can perfectly recover violation_type → recover is_citation = (violation_type == "Citation").

The fix: Replaced violation_type + violation_type_top with arrest_type_letter in TE_CATEGORICALS. arrest_type_letter has 19 distinct codes with citation rates spanning 0.32-0.92 — no perfect lookup possible.

Result: disposition_auc = 0.674 (honest). Composite +0.479 at fold 4 — realistic, not inflated.

Trade-off cost of the leak fix

Removing violation_type_te_* hurt other heads that DID get a real (non-leaky) signal from those columns:

Headv3.2.0 AUC (with leak)v3.3.0 AUC (fixed)Δ
accident0.8940.863-0.031
injury0.8810.821-0.060
disposition (NEW)1.000 (leaked)0.674 (honest)—

We accepted -0.05 to -0.10 AUC drop on 2 heads to ship an honest disposition head + close two leak vectors.

Lesson: Target encoding can leak indirectly via shared categorical bases. When you add a new prediction head, audit ALL existing target-derived features against the new target — not just the obvious ones. AUC = 1.000 is your alarm system, not a celebration.

13Future Work

What's NOT built and what's worth building next.

ML model improvements

Conformal prediction wrapper

Calibrated uncertainty intervals on every prediction. Currently model gives point predictions but no honest uncertainty. Vovk 2005 framework — model-agnostic post-hoc.

Multi-seed ensembling

3-5 seeds × 4 folds = 12-20 ensemble members. Each seed's random init produces different errors → averaging cancels noise. Expected lift: ~+0.01-0.02 composite per added seed.

Fairness audit

Disparate-impact analysis on demographic features (race entropy, reg state diversity). Currently uses cell-level demographic aggregates without auditing for systematic bias.

v3.4.0: 3-way disposition

Citation / Warning / SERO — finer-grained outcome prediction than current 2-way binary.

Known limitations / honest discussion

  • Disposition fold-0 weakness: 83% class imbalance on early data amplifies noise. Best AUC=0.890 there (suspicious — possible artifact).
  • strictness_speed_over leakage (partial): filtered by is_speed_related (different head's target). Leaky for the speed head only — flagged for v3.4.0.
  • No external alerting / health monitoring: Sentry catches errors, but no uptime monitoring on inference latency or model drift.
  • Walk-forward CV is strict but not perfect: per-cell rolling features (stops_last_7d) are computed once at the data-prep stage, not per-fold. Could leak slightly in early folds. Negligible empirically.
  • 4-fold ensemble vs full 5-fold: fold 2 skipped due to data anomaly, not architectural choice. Fixing the underlying split would let us include it.
  • Hyperparameter ablation incomplete: d_token, n_layers, dropout chosen from defaults + slim ablation. Not exhaustive grid search.
